# UI 重构方案文档

## UI Foundation / 统一设计基座

为了避免页面越做越散，当前 Web 调试工具需要先建立一套统一的 UI Foundation，而不是继续按页面各自定义字号、间距和卡片样式。

这一层的目标是：

- 让标题、正文、说明文字拥有固定层级，像 Markdown 一样可预期
- 让卡片、按钮、输入框、标签、状态徽标使用统一尺寸和圆角
- 让新页面默认继承同一套色彩、阴影、间距和动效，不再“每页一套”

### 统一令牌分层

建议整个前端样式按 4 层组织：

1. Design Tokens  
   颜色、字体、字号、行高、间距、圆角、阴影、动效时长
2. Typography Scale  
   `display / title-1 / title-2 / title-3 / body / body-sm / caption / micro`
3. Semantic Components  
   `panel / card / button / field / badge / tooltip / overlay`
4. Page Layout  
   页面级编排，只组合基础能力，不重新定义视觉语言

### 统一文字层级

建议把页面中的文字统一映射到下面这组语义层级：

- `display`
  页面主标题、Hero 标题、品牌标题
- `title-1`
  抽屉标题、模态标题、一级详情标题
- `title-2`
  面板标题、模块标题、页面内分区标题
- `title-3`
  卡片标题、列表项标题、关键对象名称
- `body`
  正文、段落、表单说明、运行态描述
- `body-sm`
  次级说明、摘要、辅助提示
- `caption`
  Eyebrow、字段标签、指标标签、阶段编号
- `micro`
  标签内小字、元信息、工具提示内短说明

### 统一组件基线

建议统一遵守以下口径：

- 输入框和主要按钮默认高度一致
- 卡片与 Panel 默认使用同一组圆角和阴影
- 状态徽标只允许成功 / 中性 / 风险三种语义色
- Tooltip 统一使用同一套触发尺寸、背景、圆角和文字大小
- 页面主间距、卡片内边距、表单栅格间距全部走 spacing token

### 当前落地约定

当前代码已经开始把这套基座沉到 `app/web/styles.css` 的 `:root` 变量中，后续新增页面或组件应优先复用这些变量，而不是直接写新的硬编码值。

后续如果继续推进，可以再补两件事：

- 已补充独立文档：[UI Token Reference](./ui-token-reference.md)
- 已建立 `panel / card / form / list / detail` 标准骨架，并开始逐页接入

## 1. 文档目的

本文档用于为 `resume-interview-workbench` 制定一套更产品化的 UI 重构方案。

当前项目已经具备完整的数据库主路径、桌面 portable 运行时、模板管理、面试会话、复习域、知识检索与后台任务能力，但前端界面仍然更接近“开发期工作台”，而不是“长期可用的桌面应用”。

这次重构的目标不是单纯换一套样式，而是完成下面三件事：

- 重新梳理信息架构，解决“所有内容堆在一个界面里”的问题
- 把高频用户操作与低频调试操作分层，提升空间利用率
- 为后续模板中心、复习系统、知识检索和桌面设置页预留清晰的产品入口

本文档默认服务于 Web 工作台与 Tauri 桌面版共用的前端主线。

## 2. 当前问题判断

### 2.1 当前界面的核心问题

从当前页面结构看，以下内容被同时堆叠在同一主界面中：

- 候选人简历导入
- 模板列表管理
- 模板编辑表单
- 面试启动入口
- 面试会话主工作区
- 计划、图谱、报告、后台任务
- observability 与 portable runtime 等调试能力

这导致了几个明显问题：

1. 用户任务不聚焦  
   “导入简历”“编辑模板”“启动面试”“查看面试过程”“清理本地数据”属于完全不同层级的任务，不应该出现在同一视图中争夺注意力。

2. 高价值空间被低频内容占用  
   面试会话页最宝贵的是对话区、当前线程、复盘与计划，但现在大量空间被模板编辑区和调试面板挤占。

3. 信息密度高但缺乏层次  
   当前页面有很多 panel，但 panel 之间缺乏明确层级关系，更像“所有功能同时打开”的工程控制台。

4. 用户态与开发者态混杂  
   `observability`、`background jobs`、`portable runtime` 这些能力很重要，但不应成为普通使用路径中的默认主内容。

5. 后续扩展空间不足  
   如果继续沿用当前单页堆叠模式，后续接入复习页、知识库页、设置页、会话历史页时，界面会进一步失控。

### 2.2 根本判断

当前 UI 的主要矛盾不是视觉风格，而是信息架构还停留在“调试工作台”阶段。

因此本次重构应从：

- `single-screen workbench`

演进为：

- `navigation-first desktop app`

也就是：

- 先建立应用级导航与页面分层
- 再把具体模块各自收口到最适合自己的工作区

## 3. 重构目标

### 3.1 核心目标

1. 建立清晰的一级导航  
   让用户明确知道“我现在在哪个模块”，而不是面对一个巨大的混合工作台。

2. 把不同任务拆到不同页面  
   模板管理、模板编辑、开始面试、会话进行中、复习、设置，不再互相挤压。

3. 强化主路径  
   把“导入简历 -> 选择模板 -> 开始面试 -> 复盘 -> 复习”做成自然顺滑的主链路。

4. 降低学习成本  
   首次使用者应该能快速理解产品结构，而不是需要先理解整个工程系统。

5. 兼顾用户态与开发者态  
   保留调试与观测能力，但默认只在开发者模式或专门入口中出现。

### 3.2 非目标

以下内容不作为本次 UI 重构的直接目标：

- 一次性重写全部前端状态管理
- 一次性引入大型前端框架升级
- 一次性做完全部动画、主题系统与视觉品牌升级
- 在首轮重构中把所有后端能力都做成完整产品页

首轮重构更强调“结构先对”，然后再逐步补细节。

## 4. 设计原则

### 4.1 应用分层优先于卡片堆叠

不要继续通过增加 panel 来解决功能增长问题。

优先采用：

- 一级导航
- 页面切换
- 详情页
- 抽屉 / 模态
- Inspector 侧栏

而不是继续把所有东西堆在同一屏。

### 4.2 高频任务放主界面，低频任务放次层

高频任务：

- 导入简历
- 选择模板
- 开始面试
- 回答问题
- 查看复盘
- 进入复习

低频任务：

- 运行时清理
- 后台任务观测
- observability
- 目录查看
- 调试态图谱细节

低频任务不应默认占据主视图。

### 4.3 用户态与开发者态明确分层

普通用户应该看到的是：

- 工作台
- 模板
- 面试
- 复习
- 设置

开发者才需要默认关注：

- jobs
- observability
- runtime health
- graph 原始细节

这两类信息应通过模式切换或独立入口分离。

### 4.4 会话页必须以“对话”为中心

面试会话进行中，核心主角永远是：

- 当前问题
- 当前回答
- 线程上下文
- 当前阶段

因此会话页必须围绕对话展开，而不是让对话区成为多个边栏之间的一块普通 panel。

### 4.5 表单编辑页与资源管理页分开

模板列表页解决“找哪个模板”的问题。  
模板编辑页解决“怎么编辑这个模板”的问题。

这两种任务不应混在一个面板里。

### 4.6 默认内容保持中性与可发布

UI 空状态、占位文案、示例模板、演示数据必须保持中性，不使用真实公司、产品、人员或其他可识别信息。

这既是发布合规要求，也是桌面 clean portable 策略的一部分。

## 5. 目标信息架构

建议将应用重构为 6 个一级模块：

1. 工作台
2. 开始面试
3. 模板
4. 复习
5. 知识库
6. 设置

### 5.1 一级模块说明

| 模块 | 目标 | 默认用户 |
| --- | --- | --- |
| 工作台 | 展示总览、最近活动、待处理事项 | 所有用户 |
| 开始面试 | 创建新的面试 session | 所有用户 |
| 模板 | 管理模板资产与编辑模板 | 高频配置用户 |
| 复习 | 查看弱项、复习集、attempts、recommendations | 高频复盘用户 |
| 知识库 | 浏览题库、知识文档、相似召回结果 | 进阶用户 |
| 设置 | 管理 provider、数据目录、runtime、开发者模式 | 所有用户 / 开发者 |

### 5.2 建议导航结构

建议采用桌面应用常见的导航方式：

- 左侧固定主导航 rail
- 顶部为当前页面标题与操作区
- 中间为页面主工作区
- 右侧 inspector 按页面需要出现

推荐一级导航顺序：

1. 工作台
2. 开始面试
3. 模板
4. 复习
5. 知识库
6. 设置

如果后续需要开发者入口，可在设置页中再进入：

- 开发者工具
- 运行时观测
- 任务与日志

## 6. 页面拆分方案

### 6.1 工作台首页

#### 页面目标

作为应用首页，只做总览与轻入口，不承载重编辑。

#### 建议内容

- 当前简历状态卡片
- 最近使用模板
- 最近 session
- 待复习弱项摘要
- 后台任务摘要
- 快捷入口

#### 适合放在这里的操作

- 导入简历
- 快速开始一次新面试
- 打开最近模板
- 打开最近 session
- 查看待复习项

#### 不适合放在这里的内容

- 长模板表单
- 全量会话详情
- observability
- runtime danger zone

### 6.2 开始面试页

#### 页面目标

将“启动一次新面试”做成专门页面，避免和模板编辑混在一起。

#### 建议布局

页面中部以 3 个步骤卡呈现：

1. 选择简历工作区状态
2. 选择模板
3. 选择面试基座与启动参数

#### 核心字段

- 当前简历状态
- 模板选择
- 角色基座
- 岗位基座
- web search 开关
- 备注

#### 页面结果

- 点击“开始面试”后进入会话页

### 6.3 模板中心页

#### 页面目标

承载模板资产的浏览、筛选、复制、归档和新建。

#### 建议布局

- 顶部：标题、搜索框、筛选器、新建按钮
- 左侧：模板列表 / 分组
- 右侧：模板摘要卡片

#### 列表字段建议

- 模板名
- 岗位方向
- 更新时间
- 最近使用
- 状态

#### 支持操作

- 新建
- 复制
- 编辑
- 归档
- 删除

### 6.4 模板编辑页

#### 页面目标

把当前侧栏里的模板表单升级为独立编辑体验。

#### 建议分区

1. 基本信息
   - 模板名称
   - 公司/团队名称
   - 岗位方向

2. 背景信息
   - 公司/团队介绍
   - 岗位介绍
   - 其他上下文

3. 面试官设定
   - 面试官角色名
   - 默认风格

4. 专家模式
   - 角色基座
   - 岗位基座
   - 未来可接默认策略参数

5. 预览区域
   - 模板摘要
   - 适用方向
   - 生成效果预览

#### 页面交互建议

- 顶部固定保存栏
- 支持“保存”“另存为副本”“返回模板中心”
- 未保存离开时提示

### 6.5 面试会话页

#### 页面目标

成为真正的“面试主战场”，只关注会话过程。

#### 建议布局

- 顶部：session header
- 中间主栏：对话区
- 左侧窄栏：阶段、线程、当前概览
- 右侧 inspector：按 tab 切换

#### 右侧 inspector 建议 tab

- 计划
- 图谱
- 复盘
- 线程

开发者模式下再追加：

- jobs
- observability
- runtime

#### 页面重点

- 对话区应成为视觉中心
- 回答输入区应稳定固定在底部
- 当前问题信息应比历史辅助信息更突出

### 6.6 复习页

#### 页面目标

把当前 review 域正式产品化。

#### 建议内容

- review item 列表
- 复习集列表
- recommendation 列表
- attempt 历史

#### 可拆成二级页

- 复习总览
- review item 详情
- review set 详情

### 6.7 知识库页

#### 页面目标

统一承接题库、知识文档和检索结果，而不是散落在多个 service 背后。

#### 建议内容

- 题库列表
- 标签 / 主题筛选
- 文档详情
- 相似检索结果
- usage stats

### 6.8 设置页

#### 页面目标

把环境配置与运行时能力移出主工作台。

#### 建议分区

1. 模型配置
2. Embedding 配置
3. 数据目录与本地工作区
4. portable runtime 清理
5. 导入导出
6. 开发者模式

#### danger zone

危险操作统一放在设置页底部：

- 清理缓存
- 清理日志
- 重置本地设置
- 删除全部本地数据

## 7. 组件分层建议

建议把前端组件拆成四层：

### 7.1 App Shell 层

负责：

- 顶层布局
- 左侧导航
- 顶部标题栏
- 页面容器

### 7.2 Page 层

负责：

- 每个一级页面的整体结构
- 页面级数据拉取与局部状态

建议页面模块：

- `dashboard-page`
- `start-session-page`
- `template-list-page`
- `template-editor-page`
- `session-page`
- `review-page`
- `knowledge-page`
- `settings-page`

### 7.3 Feature 层

负责：

- 模板列表
- 模板表单
- session header
- conversation timeline
- plan inspector
- report inspector
- runtime panel

### 7.4 Primitive 层

负责：

- button
- input
- select
- card
- tab
- empty state
- drawer
- modal
- badge

这样后续视觉升级时，不需要重复修改页面内部结构。

## 8. 状态与路由建议

### 8.1 路由建议

当前项目仍是轻量服务端 + 原生前端结构，首轮不必引入复杂路由框架，但建议至少建立页面级 view state：

- `dashboard`
- `session-start`
- `templates`
- `template-editor`
- `session`
- `review`
- `knowledge`
- `settings`

可先用以下方式实现：

- URL query + history API
- 本地 `state.currentView`
- 页面切换时局部渲染

### 8.2 状态拆分建议

建议按模块拆：

- `appState`
- `bootstrapState`
- `templateState`
- `sessionState`
- `reviewState`
- `knowledgeState`
- `settingsState`
- `developerState`

首轮可以不完全重构，但至少避免把所有页面状态继续堆进一个总对象里。

## 9. 视觉方向建议

### 9.1 总体方向

当前视觉基调已经有一定质感，但布局仍偏“信息墙”。

建议朝下面方向收口：

- 桌面应用感
- 单任务聚焦
- 明确的主次层级
- 更稳定的页面节奏

### 9.2 推荐视觉特征

- 左侧深色或低饱和导航 rail
- 页面主区域使用大块留白
- 表单页采用 section 分组而不是长卡片堆叠
- 会话页采用固定输入区和可切换 inspector
- 设置页采用 list section + detail 结构

### 9.3 参考方向

可参考的交互气质：

- Notion：页面层级清晰
- Linear：单任务聚焦
- Slack / Discord 桌面端：导航层明确
- Apple 设置类应用：分组简洁

不是照搬样式，而是学习其结构组织方式。

## 10. 首轮实施范围建议

建议第一轮不要同时重写全部页面，而是优先完成结构收口。

### Phase A：应用壳与导航

目标：

- 建立左侧一级导航
- 引入页面级 view state
- 把当前单页拆成多个页面容器

完成标准：

- 已可以在不同主页面之间切换
- 原首页不再承载全部功能

### Phase B：模板模块拆分

目标：

- 建立模板中心页
- 建立模板编辑页
- 从首页移除长模板表单

完成标准：

- 模板管理与模板编辑已分离
- 开始面试入口不再依赖侧栏大表单

### Phase C：会话页重构

目标：

- 会话页只保留会话相关内容
- 对话区成为视觉中心
- inspector tab 化

完成标准：

- 会话页默认不再展示 runtime / observability
- 用户态与调试态开始分层

### Phase D：设置与开发者入口

目标：

- 收口 provider / runtime / cleanup / developer mode
- 将 portable runtime 与 observability 移出主工作台

完成标准：

- 高风险与低频操作不再出现在主工作区

### Phase E：复习与知识库产品化

目标：

- 为 review / knowledge 建立正式页面

完成标准：

- review 与 knowledge 不再只是底层能力，而是可访问的产品模块

## 11. 推荐落地顺序

建议按下面顺序实施：

1. 先做应用壳与导航，不先追求完美视觉
2. 再拆模板中心与模板编辑页
3. 再重构开始面试页
4. 再重构会话页
5. 再把设置 / 开发者入口收口
6. 最后补复习页与知识库页

原因是：

- 先把架构搭好，后面每个模块才有稳定归宿
- 不必一开始就重写所有页面逻辑
- 可以每一阶段都看到明显变化

## 13. 验收标准

如果首轮 UI 重构完成，应该能达到下面效果：

1. 首页不再是所有功能的堆叠面板
2. 模板管理与模板编辑已经分离
3. 会话页聚焦于对话与会话过程
4. 设置与开发者能力有独立归宿
5. 普通用户默认看不到大量运维与调试信息
6. 后续新增复习与知识库页面时，不需要再破坏主工作台结构

## 14. 最终结论

这次 UI 重构的关键，不是“换个更好看的样式”，而是把产品从“工程控制台”升级成“分层明确的桌面应用”。

一句话概括就是：

**先做应用级导航和页面分层，再让模板、面试、复习、知识库、设置各自回到最合适的界面里。**

这样后续无论是 Web 使用形态还是 Tauri portable 桌面形态，都会更稳定、更清晰，也更适合持续演进。
