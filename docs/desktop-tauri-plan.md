# Tauri 桌面化落地方案

## 1. 文档目的

本文档用于定义 `resume-interview-workbench` 的桌面化实现路线。

目标不是重写现有系统，而是在保留当前数据库与业务架构的前提下，把项目逐步演进为：

- 用户双击即可启动
- 用户不需要手动安装 Docker
- 用户不需要手动准备 PostgreSQL 运行环境
- 数据库、服务端、前端窗口由桌面应用自动托管

本文档采用 MVP 方式拆分阶段，确保每一阶段都：

- 有清晰目标
- 有可见变化
- 有可验收结果
- 有明确回退边界

---

## 2. 当前前提

当前仓库已经完成数据库主线收尾，核心前提如下：

- `PostgreSQL + pgvector` 已是默认业务真相源
- 默认运行模式为 `database_only`
- 运行态、题库、review、knowledge、background jobs 已进入数据库主路径
- 文件层已降级为 `debug / export / backfill`
- 本地开发阶段仍推荐保留 Docker 作为数据库开发环境承载方式

这意味着桌面化的重点不是“去掉数据库”，而是：

**把数据库和服务托管能力从开发者手工操作，转变为桌面应用自动管理。**

---

## 3. 目标与非目标

### 3.1 目标

- 保留当前 `PostgreSQL + pgvector` 数据架构
- 让桌面版用户不再接触 Docker、`db:migrate`、`.env` 手工配置
- 保留当前 Node 服务端和 Web 工作台，先做最小改造
- 让每一阶段都能被独立演示、验证和回退
- 最终形成 Windows 优先的桌面 portable 发布形态

### 3.2 非目标

- 第一阶段不重写现有 Node 服务为 Rust
- 第一阶段不追求真正意义上的“单 exe 打包一切”
- 第一阶段不改变主数据库选型
- 第一阶段不为了桌面化而回退数据库架构

---

## 4. 技术路线结论

### 4.1 为什么选择 Tauri

在当前项目阶段，`Tauri` 比 `Electron` 更适合做正式桌面化主线，原因如下：

- 更适合作为轻量桌面壳，承载现有 Web 前端
- 可以通过 sidecar 方式托管外部可执行文件，适合启动本地服务与数据库进程
- 更符合“正式桌面应用”而不是“开发工具壳”的产品形态
- 便于后续统一做进程管理、日志收集、路径管理与升级

### 4.2 当前推荐架构

推荐采用：

**Tauri 桌面壳 + 现有 Web 前端 + 现有 Node 服务 + 本地托管 PostgreSQL**

分层职责如下：

| 层 | 职责 |
| --- | --- |
| Tauri 壳 | 窗口管理、进程编排、路径管理、首次初始化、日志汇总、升级入口 |
| Node 服务 | 继续承载当前 API、仓储层、任务系统、检索逻辑、运行态 |
| PostgreSQL | 继续承载业务真相源、结构化读模型、题库、review、knowledge、jobs |
| Web UI | 继续承载工作台前端界面 |

### 4.3 用户视角与工程视角

#### 用户视角

- 解压 portable 包
- 双击启动
- 首次启动自动初始化
- 直接进入工作台

#### 工程视角

应用内部仍然会自动执行：

1. 解析应用目录与用户数据目录
2. 检查数据库数据目录是否存在
3. 必要时初始化 PostgreSQL data dir
4. 启动本地 PostgreSQL 进程
5. 检查数据库 readiness
6. 执行 migration
7. 启动本地服务端
8. 打开桌面窗口并加载 UI

## 5. 发布形态结论

### 5.1 我们追求的不是“没有数据库”

本项目桌面化的正确目标不是删除数据库，而是让数据库对用户“无感存在”。

因此发布形态应理解为：

- 对用户：像一个桌面应用
- 对系统：仍然有数据库、服务端、迁移与后台任务

### 5.2 我们追求的也不一定是真单文件

更现实的目标是：

- 一个桌面 portable 包
- 解压后即可得到一个可运行的桌面应用
- 应用首次启动时把必要运行时写入用户数据目录
- 后台自动拉起数据库和服务

也就是说，用户体验可以是“双击即用”，但技术实现通常是“portable 包 + 应用数据目录 + 本地进程托管”。

## 6. 目录与运行时规划

### 6.1 推荐目录

Windows 推荐目录如下：

| 类型 | 建议位置 | 用途 |
| --- | --- | --- |
| 程序目录 | 用户解压后的 portable 目录 | 放桌面应用主程序、受控 sidecar 与静态资源 |
| 用户数据目录 | `%LocalAppData%/ResumeInterviewWorkbench/` | 放数据库数据、日志、导出、缓存 |
| 数据库目录 | `%LocalAppData%/ResumeInterviewWorkbench/postgres/` | PostgreSQL data dir |
| 日志目录 | `%LocalAppData%/ResumeInterviewWorkbench/logs/` | 桌面壳、服务端、数据库日志 |
| 导出目录 | `%LocalAppData%/ResumeInterviewWorkbench/exports/` | 调试导出与备份 |

### 6.1.1 程序内数据清理约定

由于当前路线采用 portable 分发而不是安装/卸载模型，需要提前区分“可安全清理的缓存数据”和“默认不应直接删除的用户业务数据”，并将清理能力做成程序内按钮或设置页动作。

推荐按下面原则处理：

| 目录 | 默认清理行为 | 说明 |
| --- | --- | --- |
| `cache/` | 应清理 | 纯缓存目录，可由程序内“清理缓存”按钮直接删除 |
| `tmp/` | 应清理 | 临时文件目录，可由程序内“清理缓存”按钮直接删除 |
| `logs/` | 应清理 | 日志目录默认可清理，避免长期残留大量排障文件 |
| `exports/` | 可选清理 | 若目录只承载应用自动生成的导出，可由程序内提供单独清理入口 |
| `config/` | 可选清理 | 若只包含本地偏好配置，可作为“重置本地设置”能力的一部分 |
| `data/postgres/` | 默认不直接清理 | 数据库目录承载正式业务数据，不能默认静默删除 |

推荐程序内清理策略：

- 提供“清理缓存”按钮，默认清理 `cache/`、`tmp/`、`logs/`
- 提供“清理导出与本地设置”类可选按钮，覆盖 `exports/`、`config/`
- 提供“删除全部本地数据”危险操作，覆盖 `data/postgres/`，但必须显式确认

这样可以同时满足两件事：

- 用户删除程序目录后，不会因为缓存残留而造成混乱
- 用户不会因为一次误操作而意外丢失完整面试数据、题库资产和复习记录

推荐最小功能按钮集合：

- `清理缓存`
- `清理日志`
- `重置本地设置`
- `删除全部本地数据`

其中“删除全部本地数据”必须是显式确认路径，不能做成默认行为。

### 6.2 运行时端口

建议本地仍保留 HTTP 服务模型：

- PostgreSQL：本地环回地址，如 `127.0.0.1:5432` 或动态端口
- Node 服务：本地环回地址，如 `127.0.0.1:3000` 或动态端口

桌面窗口加载本地服务页面，而不是一开始就把所有逻辑改写到 Rust 内嵌命令中。

这能最大限度复用现有系统。

## 7. MVP 分阶段实施方案

以下阶段按“最快验证桌面化主线”设计，而不是按“最完美最终形态”设计。

### MVP 0：桌面壳验证

#### 目标

验证 Tauri 可以作为本项目桌面容器，并成功加载当前工作台页面。

#### 实现范围

- 初始化 `src-tauri/`
- 建立最小 Tauri 窗口
- 开发模式下直接指向现有本地 Web 服务
- 验证基础窗口生命周期、图标、应用名、开发脚本

#### 可见变化

- 仓库出现 Tauri 桌面工程
- 本地可以通过一条命令拉起桌面窗口
- 页面不再只通过浏览器访问，也可以通过桌面窗口访问

#### 交付物

- Tauri 工程骨架
- 开发态桌面启动脚本
- 最小桌面 README 或文档说明

#### 验收标准

- 本地开发机可打开桌面窗口
- 窗口可正确加载当前 UI
- 关闭桌面窗口后不会残留异常前台进程

#### 回退边界

- 只增加桌面壳，不影响现有 Web 项目运行

---

### MVP 1：桌面壳自动托管现有服务

#### 目标

让 Tauri 在开发态自动拉起当前 Node 服务，减少用户和开发者切换浏览器的依赖。

#### 实现范围

- Tauri 启动时自动检查服务是否运行
- 未运行时自动拉起本地 Node 服务
- 窗口关闭时按策略停止或释放该服务进程
- 收集桌面壳与服务端启动日志

#### 可见变化

- 启动桌面应用时，不需要手动先跑 `npm run dev`
- 桌面窗口会等待本地服务 ready 后再展示页面

#### 交付物

- 服务启动管理器
- readiness 检查逻辑
- 启动失败提示页或错误对话框

#### 验收标准

- 一条命令可启动桌面壳与服务
- 服务启动失败时有清晰错误提示
- 重复启动不会产生多个冲突服务实例

#### 回退边界

- 服务编排失败时仍可回退为手动启动 Web 模式

---

### MVP 2：桌面化本地数据库托管

#### 目标

去掉用户对 Docker 的依赖，让桌面应用自动托管本地 PostgreSQL。

#### 实现范围

- 引入受控 PostgreSQL 运行时分发方式
- 首次启动自动初始化 data dir
- 自动启动 PostgreSQL 进程
- 自动执行数据库 readiness 检查
- 自动执行 migration
- 自动写入桌面版数据库连接配置

#### 可见变化

- 用户不再需要安装和启动 Docker
- 首次启动可能出现“正在初始化本地数据环境”的提示
- 后续启动直接进入工作台

#### 交付物

- PostgreSQL 管理器
- 首次初始化逻辑
- migration 托管逻辑
- 桌面版运行日志

#### 验收标准

- 全新机器解压 portable 包后可以首次完成初始化
- 无 Docker 环境下也可正常使用
- 重启应用后能稳定复用既有数据库

#### 回退边界

- 开发环境仍保留 Docker 流程
- 桌面版数据库托管失败时可定位日志并手工排障

---

### MVP 3：桌面 portable 发布与用户无感启动

#### 目标

形成对测试与早期用户友好的 portable 发布版本。

#### 实现范围

- 输出 Windows portable 包
- 配置应用图标、名称、版本信息
- 首次启动引导
- 自动打开主窗口
- 补齐应用数据目录、日志目录、导出目录初始化
- 补齐程序内数据清理按钮与危险操作确认流程

#### 可见变化

- 用户拿到 portable 包后解压即可运行
- 双击程序即可打开
- 不再需要命令行参与主流程

#### 交付物

- Windows portable 包
- 首次启动初始化界面
- 基础错误提示与恢复引导
- 程序内数据清理策略说明

#### 验收标准

- 测试机可完成 portable 解压、首次初始化、正常启动、退出再启动
- 替换程序目录升级版本时不会直接破坏现有数据目录
- 程序内 `cache/`、`tmp/`、`logs/` 等可清理目录符合策略预期
- “删除全部本地数据”操作具备明确确认与可预期结果

#### 回退边界

- portable 发布失败时，项目仍可通过现有 Web 形态继续使用

---

### MVP 4：稳定性与恢复闭环

#### 目标

把桌面版从“能跑”升级为“可长期使用”。

#### 实现范围

- 进程异常退出检测与恢复
- 数据库不可达恢复提示
- migration 失败恢复提示
- 日志汇总入口
- 导出日志、导出数据库备份、恢复入口

#### 可见变化

- 出现故障时，用户不再只看到白屏或启动失败
- 应用内可以看到更明确的故障说明和导出入口

#### 交付物

- 桌面健康状态面板
- 错误恢复流程
- 备份与导出入口

#### 验收标准

- 故意制造数据库不可达、端口占用、迁移失败等场景时可被正确提示
- 支持导出最小排障材料

#### 回退边界

- 即使桌面恢复逻辑不完善，也不能破坏用户已有数据库目录

---

### MVP 5：产品化增强

#### 目标

把桌面版补齐到长期分发形态。

#### 实现范围

- 自动更新策略
- 版本升级迁移策略
- 多窗口或辅助工具窗口
- 系统托盘、最小化行为
- 更完整的设置页

#### 可见变化

- 用户升级路径更顺滑
- 桌面行为更接近正式客户端

#### 交付物

- 更新策略文档
- 升级测试矩阵
- 桌面设置页

#### 验收标准

- 从旧版本升级到新版本不会破坏现有数据
- 核心升级场景可重复验证

---

## 8. 每阶段如何“能看见变化”

为了确保每一阶段不是纯底层改造，建议每个阶段都至少提供一个“可被展示的变化入口”。

| 阶段 | 推荐可见入口 |
| --- | --- |
| MVP 0 | 新增桌面窗口启动命令与窗口截图 |
| MVP 1 | 桌面应用一键拉起本地服务 |
| MVP 2 | 无 Docker 情况下成功启动并完成初始化 |
| MVP 3 | portable 包与首次启动体验 |
| MVP 4 | 故障恢复页、日志导出、健康状态入口 |
| MVP 5 | 升级体验、设置页、托盘等产品化能力 |

额外建议：

- 每阶段结束都补一段短视频或截图
- 每阶段结束都更新 README 和本文件状态
- 每阶段都做一个最小演示脚本

## 9. 代码改造建议

### 9.1 新增目录建议

```text
src-tauri/
  capabilities/
  icons/
  src/
    main.rs
    bootstrap/
    process/
    paths/
    logging/
```

### 9.2 建议增加的桌面层职责模块

- `DesktopBootstrapService`
  负责桌面启动主流程编排
- `PostgresProcessManager`
  负责 PostgreSQL 初始化、启动、停止、健康检查
- `BackendProcessManager`
  负责 Node 服务启动、停止、ready 检查
- `DesktopPathService`
  负责应用目录、日志目录、数据库目录定位
- `DesktopHealthService`
  负责汇总桌面启动健康状态

### 9.3 对现有 Node 项目的要求

当前 Node 服务端需要逐步补强这些能力，便于桌面化托管：

- 更明确的健康检查入口
- 更明确的 ready 信号
- 更稳定的退出语义
- 更清晰的运行日志归集
- 更明确的 migration 托管边界

## 10. 风险与应对

### 10.1 风险：桌面化过程中引入过早重写

应对：

- 桌面化第一原则是复用现有业务系统
- 先做托管，不做重写

### 10.2 风险：PostgreSQL 分发与初始化复杂度高

应对：

- 先做 Windows 优先
- 先控制支持平台，不追求一次覆盖所有系统
- 先跑通“首次初始化 + 正常启动 + 正常升级”三条主路径

### 10.3 风险：桌面壳、数据库、服务端三方日志分散

应对：

- 从 MVP 1 开始就建立统一日志目录
- 从 MVP 4 开始建立用户可见导出入口

### 10.4 风险：阶段太大，无法快速看到结果

应对：

- 严格按 MVP 分段
- 每段都必须有外显变化
- 每段都可独立提交、演示和回退

---

## 11. 推荐实施顺序

建议按下面顺序推进：

1. 先做 `MVP 0`，验证 Tauri 壳和窗口形态
2. 再做 `MVP 1`，把现有 Node 服务编排进桌面启动链路
3. 然后做 `MVP 2`，去掉用户对 Docker 的依赖
4. 再做 `MVP 3`，输出真正可交付的 portable 包
5. 最后补 `MVP 4` 和 `MVP 5`，完成稳定性和产品化收尾

这个顺序的核心好处是：

- 每一步都有看得见的收益
- 不会因为一次做太多而失控
- 随时可以停在某个“已经可用”的阶段

---

## 12. 当前建议结论

如果我们的目标是：

- 保留当前数据库与知识检索架构
- 不让用户再安装 Docker 或手工准备环境
- 让用户双击即可进入工作台

那么最合理的桌面化主线就是：

**Tauri 桌面壳 + 现有 Node 服务 + 本地托管 PostgreSQL + 渐进式 MVP 迭代**

这条路线的关键不是“把系统改成 Rust 项目”，而是：

**先把桌面应用做成可靠的运行时托管器，再逐步把用户感知收敛为一个正式客户端。**
